设计令牌：让你的 CSS 不再失控

子不语 ·2026-06-12 ·阅读 8 分钟 ·浏览 2.1k

每一个长期维护的前端项目，几乎都会经历同一种缓慢的失序：起初只是为某个按钮加了一种「稍微深一点的红」，半年后样式表里躺着十几种近似却又不完全相同的红色。颜色、字号、间距、圆角——这些本该收敛的视觉决策，悄悄地各自繁殖。设计令牌（Design Tokens）正是为了把这种熵增重新关进笼子里。

它的思路并不复杂：把所有视觉决策抽象成有名字的变量，让设计与实现共享同一套词汇。但真正难的不是定义令牌，而是如何组织它们——让令牌既能保持一致，又能在需要时灵活地被覆盖。

问题：CSS 是怎么失控的

失控很少是一次性的崩塌，而是无数次「就这一次」累积的结果。设计稿里出现了一个新的强调色，开发者顺手写下十六进制值；下个迭代换了设计师，又带来一套审美略有差异的灰阶。没有人做错什么，但系统整体却越来越难以维护。

当我打开一个运行了两年的项目，用脚本统计实际用到的样式值时，结果往往触目惊心。这些数字背后是真实的认知负担：每一次新需求，开发者都要在一堆相似选项里做无意义的抉择。

  • 项目里散落着 17 种强调色，其中至少 9 种肉眼几乎无法区分；
  • 字号多达 15 种，从 13px 到 32px 之间挤满了没有规律的数值；
  • 圆角有 8 种，4px、5px、6px 同时存在，没人记得为什么。

这些值之所以失控，不是因为它们「太多」，而是因为它们没有名字、没有出处、没有层级。一个孤立的 #C8453B 不携带任何意图，而一个叫 var(--color-primary) 的令牌，才开始承载语义。

三层令牌架构

收敛的关键，是把令牌分成职责清晰的三层。每一层只解决一个问题，彼此通过引用串联——上层永远不直接写死值，而是指向下一层。这样既保证了一致性，又保留了局部覆盖的弹性。

原始层 Primitive

最底层是不带任何语义的「原材料」：一整套色阶、一把字号刻度、一组间距单位。它们只描述「是什么」，不描述「用在哪」。比如 --red-500: #C8453B，它只是红色色阶里的一档，本身不关心自己将被用作主色还是警告色。

语义层 Semantic

语义层把原始值映射到「用途」。--color-primary 指向某一档红，--color-danger 指向另一档。当品牌升级、主色从朱砂改为绛紫时，你只需要改动语义层的这一处指向，整个产品随之更新——而原始色阶纹丝不动。

组件别名层 Alias

最上层是面向具体组件的别名，如 --button-bg、--card-border。它默认引用语义层，但允许某个组件在不破坏全局的前提下做局部微调。这一层是「逃生舱口」：既享受系统的统一，又不被系统绑死。

「好设计是从内容里长出来的，不是套出来的。」

—— 一条贴在显示器边上的便签

令牌系统也是如此。它不该是一份从别处抄来的、悬浮在项目之上的规范，而应当从你真实使用过的颜色与字号里提炼出来。下面是一份精简到极致的语义层定义，足以支撑一个小型项目的昼夜双主题：

/* 语义层：把原始色阶映射到用途 */
:root {
  --color-primary:     var(--red-500);
  --color-bg:          var(--paper-50);
  --color-text:        var(--ink-900);
  --color-border:      var(--ink-100);
}
[data-theme="dark"] {
  --color-primary:     var(--red-300);
  --color-bg:          var(--ink-950);
}CSS

组件里从此不再出现任何裸色值，只引用语义令牌——例如按钮背景统一写成 var(--color-primary)。切换主题时，你改的永远是令牌指向，而不是组件本身。

迁移：零破坏地替换

已经失控的老项目，也不必推倒重来。令牌迁移可以做到对线上零破坏——核心是先建立映射，再逐步收口，最后才清理。

  1. 先建底座：不动任何现有样式，先把原始层与语义层定义好，让新令牌与旧值并存、互不干扰。
  2. 逐文件替换：以组件为单位，把裸色值、裸字号替换成对应令牌；每替换一个就回归一次，确保视觉无变化。
  3. 收口与清理：当裸值不再新增，用 lint 规则禁止直接写颜色值，最后删掉无人引用的死令牌。

当这套流程跑完，你会发现真正改变的不只是样式表，而是团队讨论视觉时的语言——从「把那个红改深一点」变成「primary 提一档」。令牌的终点，从来不是变量本身，而是一种被共享的、可被理解的秩序。

设计令牌：让你的 CSS 不再失控

子不语 · 2026-06-12 · 阅读 8 分钟 · 浏览 2.1k

一个项目刚起步时，CSS 总是很干净：几个颜色、几号字、一致的圆角。可半年之后，你打开样式表，发现里面藏着十几种几乎一样的蓝色、一堆魔法数字般的字号，以及没人记得为什么存在的间距。这不是某个人的疏忽，而是缺少一层「约定」——也就是设计令牌（Design Tokens）。

本文记录我把一个失控的样式系统收敛到三层令牌架构的过程：它解决了什么、怎么分层、以及如何零破坏地迁移。

01问题：CSS 是怎么失控的

失控很少是一次性发生的。它来自无数次「就先这样吧」：设计稿里多了一个偏暖的蓝，于是 CSS 里多了一个 #2a6cf0；某个按钮需要小一号字，于是出现了 15px。每一次都合理，累积起来却是一场灾难。

当我真正去审计这个项目时，统计结果让人尴尬——同一套界面里，竟然散落着这么多本应统一的值：

  • 17 种强调色（其中至少 9 种是肉眼几乎无法区分的「蓝」）
  • 15 种字号（从 11px 到 34px，没有任何比例规律）
  • 8 种圆角（3px / 4px / 5px / 6px … 全凭手感）

02三层令牌架构

收敛的核心思路，是在「具体的值」和「使用它的地方」之间插入抽象层，让组件永远不直接接触颜色码或像素值。我把它拆成三层。

原始层 / Primitive

最底层是一张调色板与刻度表：纯粹的事实，没有语义。比如 --blue-600: #2563EB、--neutral-900: #171717、--text-lg: 18px。这一层数量有限、命名机械，它的唯一职责是「存在且唯一」——同一个蓝，全站只有一处定义。

语义层 / Semantic

中间层把原始值映射到意图：--color-primary、--color-text、--color-border。它回答的不是「这是什么颜色」，而是「这个颜色用来干什么」。昼夜主题切换就发生在这一层——白天 --color-primary 指向 --blue-600，夜里指向 --blue-400，组件层毫无感知。

组件别名层 / Component Alias

最上层是给具体组件的别名：--button-bg、--card-border。它默认继承语义层，但保留了「局部破例」的能力——当某个组件确实需要偏离系统时，你能在一个明确、可追踪的地方做这件事，而不是在样式深处埋一个魔法值。

令牌不是为了让颜色变少，而是为了让每一次「变化」都有迹可循。当设计要改主色，你只动一行；当它失控，你也只需审计一层。

— 写在重构 commit 里的一句话

落到代码上，语义层引用原始层、主题在语义层切换，大致是这样：

tokens.css CSS 
1/* 原始层：唯一的事实来源 */2:root {3  --blue-600: #2563EB;4  --blue-400: #5B8DEF;5  /* 语义层：映射意图 */6  --color-primary: var(--blue-600);7}8[data-theme="dark"] {9  --color-primary: var(--blue-400); /* 只在语义层换 */10}

之后，组件里再也不写颜色码——只写 var(--color-primary)。这一个习惯，几乎消灭了「又多出一个蓝」的可能性：因为根本没有地方让你顺手写下一个新的十六进制值。

03迁移：零破坏地替换

最现实的约束是：项目在跑，不能停下来重写。好在 CSS 变量天然支持渐进迁移，我用了三步：

  1. 先建好原始层与语义层令牌，但不删除任何旧值——新旧并存，互不影响。
  2. 按模块逐个替换：把硬编码的颜色、字号换成 var(--color-*)，每替换一处就视觉回归一次，确保零差异。
  3. 全部替换后，用脚本扫描残留的裸十六进制值与魔法数字，清理原始层之外的所有「事实」，并加一条 lint 规则防止回潮。

三周后，那 17 种蓝收敛成了一个 --color-primary；15 种字号变成了一把有比例的刻度尺。更重要的是，后来再改主题、改主色，都只是改一行——CSS 终于重新变得可被理解，而这正是工程的全部意义。

设计令牌：让你的 CSS 不再失控

子不语
2026-06-12·阅读 8 分钟·浏览 2.1k

每一个长寿的前端项目，几乎都会在某个深夜迎来同一个时刻：你想改一下主色，结果在十几个文件里搜 #D9743F，改完一轮，发现按钮、链接、图标边框还是各有各的橙。CSS 没有失控，是我们对颜色和尺寸的「命名权」失控了。设计令牌（Design Tokens）就是把这份命名权收回来的方法。

问题：CSS 是怎么失控的

失控很少是一次性发生的。它是一种渐进的熵增：第一个人写下 color: #D9743F，第二个人需要一个「稍微深一点的橙」，于是有了 #C76636；第三个人在做 hover 态，凭手感调出 #B85A2A。三个月后，代码库里住着十几种「橙」，没有任何一个有名字，也没有任何一个是「官方」的。

字号、圆角、间距走的是同一条路。一个组件用 15px，另一个用 15.5px，肉眼几乎看不出差别，但它们都是真实存在的、需要被维护的「魔法数字」。当我把某个真实项目里散落的值统计出来时，结果触目惊心：

  • 17 种强调色——大部分只差几个色阶，却谁也不让谁；
  • 15 种字号——从 11px 到 34px，没有可辨认的层级规律；
  • 8 种圆角——4px、6px、8px、10px……每个组件都在重新发明圆角。

三层令牌架构

解药不是「少用几个颜色」这种自律口号——自律会过期。真正可持续的做法是建立一套三层结构，让每一层只回答一个问题。

原始层（Primitive）

最底层只描述「事实」，不带任何意图。它是一张调色板和一把尺子：orange-500: #D9743F、teal-500: #2A8C82、space-4: 16px。这一层只管「世界上存在哪些值」，不管它们用在哪里。命名按色相 + 明度阶梯，机械而可预测。

语义层（Semantic）

第二层把原始值映射到「角色」上：--color-primary 指向 orange-500，--color-link 指向 teal-500。这一层是设计意图的落点——当你想把主色从橙换成红，只改这一处指向，全站跟着变。组件永远不该直接碰原始层，只跟语义层对话。

组件别名层（Component Alias）

最上层是给具体组件的「方言」：--button-bg、--card-border。默认它们继承自语义层，但保留了「就这一个组件要破例」的余地，而不必污染全局。三层之间是单向引用：组件 → 语义 → 原始，绝不反向。

令牌不是为了让颜色变少，而是为了让每一次改色都只发生在一个地方。约束的价值，从来不在限制本身，而在它让改变变得便宜。

—— 关于设计系统的一点体会

落到代码上，语义层用一组 CSS 变量承载，明暗主题各覆盖一份即可。下面是收敛后的核心片段：

tokens.css 
/* 原始层：只描述事实 */
:root {
  --orange-500: #D9743F;
  --teal-500:   #2A8C82;
  --space-4:    16px;
  /* 语义层：映射到角色 */
  --color-primary: var(--orange-500);
  --color-link:    var(--teal-500);
}

有了这套结构，组件里写的不再是某个具体颜色，而是一个意图：按钮背景永远是 var(--color-primary)，链接永远是 var(--color-link)。换肤、换主题、改品牌色，都收敛成对语义层的一次编辑。

迁移：零破坏地替换

没人能在一个 PR 里把上百个硬编码值全换掉，也不该那样做。零破坏迁移的关键，是让新旧体系在一段时间内并存：

  1. 建层不动业务。先把原始层与语义层的变量定义好，此时它们没有任何引用方，不影响现有页面。
  2. 按模块逐步替换。从最稳定、改动最少的组件开始，把硬编码值替换成 var(--color-*)，每替换一个就视觉回归一次。
  3. 用 lint 守住成果。替换完成后，加一条规则禁止在组件里直接写颜色字面量，让新代码无法再绕过令牌——熵增就此被堵在门外。

设计令牌的回报不在某一天，而在之后的每一天：当产品要换主题、当设计师说「主色再暖一点」、当你要支持暗色模式时，你改的是一处定义，而不是一场考古。CSS 从未真正失控，只是一直在等一套能让它收敛的名字。
